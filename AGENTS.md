## 架构与测试活动记录 (2026-05-30)

### 1. 单样本过拟合训练参数调整与显存优化
*   **背景**：为验证模型收敛性，前序运行在序列长度 $T=24$ 时进行单样本过拟合训练，耗时较长且存在潜在的显存溢出（OOM）隐患。
*   **修改**：将 `tests/overfit_check.py` 的训练参数调整为 $B=2, T=6$（批大小 = 2，序列长度 = 6 帧），并在可视化图片切片时采用 `min(4, T-1)` 进行防越界保护。
*   **成效**：
    *   降低了显存占用，使模型能在 8GB/16GB VRAM 显卡下正常运行。
    *   在 100 步训练中，多分支物理损失均呈收敛趋势：
        *   **Total Loss**: $12.37 \to 11.23$
        *   **Obj Loss**: $0.17 \to 0.06$（约 3 倍下降）
        *   **Box Loss**: $3.36 \to 0.19$（约 18 倍下降，检测边界框位置对齐）
        *   **Mask Loss**: $3.35 \to 0.04$（约 75 倍下降，分割掩膜重合度提升）
        *   **Ego Loss**: $0.16 \to 0.09$（姿态评估结果合理）
    *   第 100 步的最终对齐图像已被保存并归档到 Artifact 目录。
*   **W&B 记录**：配置为离线模式 (`mode="offline"`)，本地已生成离线运行数据夹，可通过 `wandb sync` 进行云端指标同步。

### 2. 深度、光流与目标框置信度量纲失配的物理 Bug 修复
*   **物理 Bug 定位与重塑**：
    1.  **深度预测全黑修复**：在 `models/tao_core.py` 几何解码模块中，预测输出键 `"depth"` 被错误传入了逆深度 `inv_depth_resized`（数值处于 $[0, 1]$ 区间），而在损失函数 `compute_physics_loss` 中，使用的是对数深度 `log_depth`。这导致对数深度梯度方向相反，产生梯度对抗。现已纠正为深度反投影：`depth = 1.0 / (inv_depth_resized + 1e-6)`。修复后，深度损失由 1.7607 降低至 0.0830，图像分层度及清晰度显著改善。
    2.  **光流估计与真值量纲差异修复**：模型原生重投影输出的光流 `flow_final` 是以原生像素 (Pixels) 为绝对单位的运动位移（在 $256 \times 256$ 分辨率下为数像素到数十像素），但系统规范约定的真值光流为除以了 128 的归一化光流 $[-1.5, 1.5]$。本方案对 `flow_final` 执行了 `flow_final_norm = flow_final * (2.0 / w)` 归一化。修复后，光流端点平均像素误差（Flow EPE px）降至 3.00 像素，光流损失降至 0.0003，预测流与真值流的空间分布一致性显著提升。
    3.  **背景杂乱边界框抑制**：先前的离线隔离跳过了 `load_yoloe_weights` 内置权重下载功能，使模型从随机初始化参数开始。现已恢复权重自动加载逻辑，并增加了 `try-except` 异常保护。成功引入 YOLOE 预训练权重后，配合 Focal Loss，在训练 100 步后，背景区域的无序候选框得到有效抑制（`Obj Loss` 降为 0.0000），预测物体框与实例真实边界对齐。

### 3. 可视化模块中绝对步数硬编码拦截的剔除与自适应动态重构
*   **物理 Bug 定位与重构**：
    1.  **基于全局步数的硬编码拦截缺陷**：前序版本为了防止前期阶段未激活追踪训练时随机初始化的绿色边界框影响画面对比，在 `visualization.py` 中硬编码了 `step >= 5000` 拦截。这一设计在改变训练参数或进行单样本过拟合验证时，会导致追踪框无法被绘制。
    2.  **动态解耦重构**：排除了依赖绝对 step 值的硬启动限制。重构了 `save_visualization` 的签名，引入 `draw_track=None` 参数。在内部通过 `step` 自适应获取当前的阶段权重配置 `w["track"]`（通过 `get_loss_weights(step)` 获得）。只有当追踪模块在当前 `global_step` 下权重非零时（`w["track"] > 0.0`），可视化才绘制追踪框，否则静默屏蔽，以自适应方式展示追踪精度。

### 4. 取消绝对步数相关的损失权重控制与真值（GT）可视化单色（蓝色）重构
*   **重构背景与物理直觉**：
    1.  **取消绝对步数损失控制**：在自监督时空物理框架中，深度、光流、不确定性、时序追踪等分支的联合训练更有利于时空物理对齐。基于硬编码步数的课程过渡容易引起超参数调整时的网络训练不稳定。
    2.  **真值框与掩膜单色化统一**：由于进行的是类别无关的物理物体发现与追踪，不再使用语义类别标签，因此真值中不同颜色的区分较为冗余，且容易增加比对复杂度。
*   **具体修改方案**：
    1.  **去除了损失调度阶段门控**：在 `utils/losses.py` 中，去除了 `STAGE_STEPS` 步数阶梯约束。将 `get_loss_weights(step)` 重构为静态返回全损失项活跃的权重字典。深度、监督光流、时序追踪、自我运动估计和特征异常检测从第 1 步起同时启动。
    2.  **取消 bounding box 损失时序过渡**：在 `utils/losses.py` 中，移除了 box 回归损失中 Smooth L1 与 GIoU 的步数过渡门控，直接在前向与反向传播中采用标准的 `giou_loss` 损失。
    3.  **真值可视化单蓝色（Blue）统一**：在 `utils/visualization.py` 中，去除了依赖 `is_dynamic` 绘制红/蓝框的代码，将 Ground Truth 图像中所有被发现实例的掩膜与边界框全部统一绘制为纯蓝色 `(255, 0, 0)`。这与预测图像（统一绘制为纯红色 `(0, 0, 255)`）形成了对比，提升了物理看板对比的直观性。同时增加了容错，当数据集缺失 `is_dynamic` 字段时依然可以正确绘制真值。

### 5. 取消骨干网络梯度的渐进解冻机制与底层优化器 Bug 修复
*   **物理架构隐患修复**：
    1.  **取消动态步数门控**：前序设计在 `trainer.py` 中使用 `self.global_step >= self.args.unfreeze_step_1` 来动态在训练中途解锁骨干网络。由于在优化器初始化后解锁的参数若未加入 optimizer 的 `param_groups`，在反向传播中将无法被实际更新。
    2.  **解耦与静态控制设计**：废除了 `train.py` 中的 `unfreeze_step_1` 与 `unfreeze_step_2` 调度，将其标记为 `[已废弃]`。骨干网络的更新权限由命令行参数 `--freeze` 在模型初始化（`__init__`）时进行静态直接控制（未传入 `--freeze` 则全程求导并装载优化器；传入 `--freeze` 则全程锁定不参与更新）。
    3.  **剔除分类标记中途掩码重写**：由于类别无关物理发现下分类损失的物理权重永久固定为 `0.0`，去除了基于 `global_step` 进行 `cls_dense` 重写覆盖 `-100` 的冗余 warmup 代码，保持端到端几何表示的整洁度。

### 6. 权重加载时 CUDA 与 CPU 跨设备张量计算的 Bug 修复
*   **物理 Bug 定位与重塑**：
    1.  **跨设备张量加法冲突**：在 `trainer.py` 的权重加载模块 `_load_yolo_weights` 中，以 `map_location="cpu"` 模式加载官方 YOLOE 预训练权重。而在偏置折叠（BN 加法吸收）步骤中，代码尝试执行 `new_sd[bn_key] = tgt[bn_key] + src_v`。此时目标模型 `tgt[bn_key]` 已迁移至 GPU（`cuda:0`），而源偏置张量 `src_v` 处于 CPU 上，导致运行时抛出非同设备计算的 `RuntimeError`。
    2.  **动态设备对齐修复**：在执行加法前添加了显式的 `.to(tgt[bn_key].device)` 动态转换，消除了设备冲突隐患，实现了 YOLO 预训练权重加载与偏置折叠。

### 7. TensorFlow 与 PyTorch 底层 CUDA Context 冲突死锁修复
*   **背景**：在训练启动时，主线程的 PyTorch 正在迁移模型至 GPU，而后台拉取 TFDS 数据集的子线程在加载 TensorFlow。两个框架并发尝试初始化底层 CUDA Context 容易触发多框架底层 CUDA 初始化冲突死锁，表现为数据拉取到一定数目后主循环永久挂起。
*   **修改**：在 `dataset.py` 的 TensorFlow 模块载入处，显式加入强制过滤逻辑：`tf.config.set_visible_devices([], 'GPU')`。
*   **成效**：对 TensorFlow 禁用 GPU，规避底层 Context 初始化冲突。数据流与 PyTorch 核心计算解耦，训练过程启动平顺。

### 8. 混合精度（FP16）训练下归一化除零引发 NaN 修复
*   **物理 Bug 定位与重塑**：在混合精度 (AMP float16) 训练下，`F.normalize(x, dim=-1)` 的默认防止除零偏置项 `eps`（为 `1e-12`）在 float16 中会发生下溢并截断为 `0.0`。一旦网络前期的 `prototypes` 或 `mask_weights` 出现全零向量或极小局部波动，归一化操作就会执行除以 `0.0`，从而使得密集 SE(3) 分配矩阵 `A` 包含 NaN。该 NaN 状态会向后传播，最终导致 Flow Loss 呈现 NaN。
*   **修复方案**：在 `models/custom_blocks.py`、`utils/geometry.py`、`models/yoloe_head.py` 等所有使用到 `F.normalize` 的地方，显式设置针对 FP16 安全的 `eps=1e-4`，防止出现零除现象，维护时空重投影流计算的数值稳定性。

### 9. 进度条动态可视化与日志精简
*   **体验重构背景**：原先的训练循环每 10 步就会向终端打印一条包含全部细粒度物理损失的极长日志，且后台 `DataBuffer` 会无间断输出当前缓冲水位，容易产生较多冗余终端输出，不利于长时间训练的指标观察。
*   **重构方案**：
    1.  **静默数据缓冲输出**：修改了 `dataset.py` 中 `AsyncDataBuffer` 的输出频率，仅在前 64 个样本载入（暖机阶段）打印进度，随后训练中途保持静默。
    2.  **进度条深度结合**：重构了 `trainer.py` 中的 `_train_epoch` 循环。使用 `tqdm.auto.tqdm` 对每个 epoch 的步骤进行进度条封装。在 `_train_chunk` 进行每 10 步的指标统计时，直接将所有精细物理损失以 Postfix 字符串的形式动态反馈到进度条右侧。使得 Colab 环境下的输出信息较为简洁，保留了进度条指示。

### 10. 首步训练耗时与光流 inf/NaN 奇点修复
*   **首步耗时定位**：在最初的几十个步骤中（从数据拉取完成到第 10 步耗时较长），原因为在 `train.py` 中启用了 `torch.backends.cudnn.benchmark = True`。对于视频/时空模型，cuDNN 会在第一次前向传播时穷举搜索最优的 3D/2D 卷积底层算子，这会产生一定的前期开销，随后的步骤会恢复正常耗时。
*   **光流 inf/NaN 奇点定位**：在第 40 步左右光流呈现 `inf`，原因为当网络预测的刚体平移把 3D 点推到了相机背后（即深度 $Z \le 0$）时，几何投影函数 `x / z` 产生除零或极小值除法。由于 $Z$ 被 clamp 到 `1e-3`，类似于 $256 / 1e-3$ 的计算超出了 float16 的表达上限 65504，导致张量溢出为 inf 并反向传播引发 NaN。
*   **修复方案**：
    1.  在 `models/custom_blocks.py` 的 `project` 投影函数中，强制将其提升至 `float32` 进行运算，将 $Z$ 截断调整为更安全的 `0.01`，并对最终算出的 `uv` 坐标硬截断到 `[-50000, 50000]`，避免了 float16 溢出。
    2.  在 `models/tao_core.py` 中，对最终输出的归一化光流 `flow_final_norm` 增加 `torch.clamp(..., -20.0, 20.0)` 的范围保护，防止个别异常像素投影在 L1 损失中产生过大梯度。
